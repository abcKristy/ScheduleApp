## Общее описание

**Schedule Application** - это полнофункциональная система для управления академическим расписанием, состоящая из серверной части (backend) на Spring Boot и мобильного приложения на Android с использованием Kotlin и Jetpack Compose. Проект предназначен для образовательных учреждений и предоставляет удобный доступ к расписанию занятий для студентов и преподавателей.

### Основной функционал
- **Просмотр расписания** для групп, преподавателей и аудиторий
- **Автоматическое обновление** данных из официальных источников
- **Оффлайн-доступ** к расписанию через кэширование
- **Гибкий поиск** с историей запросов
- **Поддержка повторяющихся событий** и исключений

### Мобильное приложение
- **Современный интерфейс** на Jetpack Compose
- **Адаптивный дизайн** с поддержкой светлой и темной темы
- **Быстрая навигация** между разделами
- **Персонализация** профиля пользователя
- **Виджеты** для быстрого доступа к расписанию

### Технические особенности
- **Многоуровневое кэширование** данных
- **Оптимизированная работа с сетью**
- **Локальное хранение** данных через Room
- **Автоматическая синхронизация**
- **Обработка ошибок** и graceful degradation

## Реализация

### Основной поток данных

#### 1. Получение расписания
```
Запрос → Контроллер → ScheduleService → Многоуровневый поиск:
    1. Память/кэш (CheckDataInMemory)
    2. База данных (ScheduleReadService) 
    3. Внешнее API (ScheduleMapper + ParserToLesson)
    4. Сохранение в БД (ScheduleWriteService)
```

#### 2. Обработка iCalendar данных
```java
// Парсинг внешнего API ответа
List<ResponseDto> response = scheduleMapper.mapToResponseDto(entities, API_URL);
List<ScheduleDto> schedule = scheduleMapper.mapToScheduleDto(response);
List<LessonEntity> lessons = scheduleMapper.parseStringData(schedule);

// Сохранение с дедупликацией
writeService.saveLessonsAndUpdateIds(lessons, response);
```

#### 3. Многоуровневое кэширование
- **Валидация сущностей** - проверка корректности входных данных
- **Проверка в БД** - поиск существующих записей
- **Внешний источник** - обновление из API при необходимости
- **Дедупликация** - устранение дубликатов при сохранении

### Ключевые особенности реализации

#### Оптимизация производительности
- **Пакетная обработка** - массовые операции с данными
- **N+1 решение** - оптимизированные запросы к БД
- **Параллельная обработка** - для больших объемов данных
- **Дедупликация** - предотвращение дублирования записей

#### Обработка ошибок
- **Глобальный exception handler** - централизованная обработка
- **Graceful degradation** - продолжение работы при частичных сбоях
- **Детальное логирование** - отслеживание всех операций

#### Транзакционность
- **Read-only транзакции** - для операций чтения
- **Write транзакции** - для операций записи
- **Разделение ответственности** - между сервисами

## API эндпоинты

### Основной эндпоинт
```http
GET /schedule/final/{entities}
```

**Параметры:**
- `entities` - список сущностей через `:` (группы, преподаватели, аудитории)

**Пример:**
```http
GET /schedule/final/ИВБО-01-20:ИВБО-02-20:Иванов И.И.:А-101
```

**Ответ:** Массив объектов `ScheduleResponseDto` с детальной информацией о занятиях.

### Формат ответа
```json
[
  {
    "discipline": "Математика",
    "lessonType": "LK",
    "startTime": "2024-01-15T09:00:00",
    "endTime": "2024-01-15T10:30:00",
    "room": "А-101",
    "teacher": "Иванов И.И.",
    "groups": ["ИВБО-01-20", "ИВБО-02-20"],
    "recurrence": {
      "frequency": "WEEKLY",
      "interval": 1,
      "until": "2024-06-15T00:00:00"
    },
    "exceptions": ["2024-03-08"]
  }
]
```

## Добавленные возможности

Этот раздел описывает ключевые улучшения и новую функциональность, добавленные в проект после базовой реализации.

### 1. Система управления семестрами

Реализована комплексная логика для работы с учебными семестрами, решающая проблему устаревания данных.

*   **Серверная часть**:
    *   **Автоматическое определение семестра**: Логика на основе даты для определения весеннего (1 февраля - 30 июня) и осеннего (1 сентября - 31 января) семестров.
    *   **Метаданные в БД**: Создана таблица `schedule_metadata` для отслеживания актуальности загруженных расписаний. Поле `semester` добавлено в сущность `Lesson`.
    *   **API проверки**: Добавлен эндпоинт `/schedule/metadata`, который сообщает клиенту, требуется ли обновление данных для конкретной группы.
    *   **Периодическая очистка**: Запланированная задача (`@Scheduled`) ежедневно удаляет занятия старше 30 дней после окончания семестра.

*   **Клиентская часть**:
    *   **Интеграция с Room**: Схема базы данных обновлена (миграция на версию 2). Добавлены поля `semester`, `cachedAt`, `expiresAt` в `ScheduleEntity` и создана таблица `CachedGroupEntity`.
    *   **Визуальная индикация**: На экране расписания отображается название текущего семестра. При его смене появляются предупреждающие баннеры (`OutdatedSemesterBanner`, `SemesterChangedBanner`).
    *   **Автообновление**: При запуске приложение проверяет семестр и, если он изменился, инициирует фоновое обновление всех кэшированных групп.

### 2. Продвинутая стратегия кэширования

Кэширование стало "умным", что значительно улучшило пользовательский опыт.

*   **Время жизни кэша (TTL)**: Данные в локальной БД автоматически считаются устаревшими через 7 дней.
*   **Фоновое обновление**: С помощью `WorkManager` реализованы воркеры (`SemesterCheckWorker`, `CacheCleanupWorker`) для фонового обновления кэша. Они приоритизируют текущую группу пользователя и работают пакетами.
*   **Управление кэшем**: Добавлен новый экран в настройках (`CacheSettingsScreen`), где можно:
    *   Увидеть объем кэша и количество сохраненных групп.
    *   Принудительно обновить все группы.
    *   Полностью очистить кэш.
    *   Включить/выключить автообновление.

### 3. Обработка сетевых ошибок и Offline-режим

Приложение стало устойчивым к проблемам с сетью.

*   **Offline-доступ**: При отсутствии интернета приложение автоматически переключается на показ данных из локального кэша.
*   **Индикация состояния**: Баннер `OfflineIndicator` информирует пользователя о работе в офлайн-режиме.
*   **Отложенная синхронизация**: Если во время поиска группы нет сети, она помечается как "ожидающая". При восстановлении соединения `NetworkMonitor` автоматически запускает загрузку данных для этой группы через `NetworkRetryWorker`.
*   **Экспоненциальная задержка**: Если API МИРЭА не отдает данные нового семестра, система не спамит запросами, а увеличивает интервал между попытками (с 7 дней до 6 часов).

### 4. Улучшения UI/UX

*   **Edge-to-Edge дизайн**: Фон приложения теперь корректно отображается под системными барами (статус-бар, панель навигации).
*   **Интерактивные элементы**:
    *   `Pull-to-Refresh` на экране расписания для ручного обновления.
    *   Кнопка принудительного обновления (`RefreshButton`) с анимацией загрузки.
    *   Свайпы для обновления и удаления элементов в истории поиска (`HistoryItemWithStatus`).
*   **Информативные баннеры**: Добавлены баннеры для различных состояний: `SummerHolidayBanner` (летние каникулы), `NewSemesterPendingBanner` (ожидание данных нового семестра).

### 5. Технический долг и оптимизация

*   **Очистка БД**: Удалены 10+ таблиц, не относящихся к проекту (`administrator`, `employer`, `multimedia`, и т.д.).
*   **API Интеграция**: Переход с нестабильного `_next/data` эндпоинта на прямой `iCal` API МИРЭА.
*   **SSL/TLS**: Настроен `RestTemplate` для работы с HTTPS API МИРЭА (решение проблемы `SSLHandshakeException`).
*   **Исправление навигации**: Устранена ошибка с `NavController`, из-за которой не работала кнопка "назад".

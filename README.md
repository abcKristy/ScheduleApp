# Schedule Service

## Описание проекта
**RESTful API сервер** для работы с расписанием занятий МИРЭА. Сервер предоставляет актуальное расписание для студентов, преподавателей и аудиторий, интегрируясь с официальным API МИРЭА и предоставляя удобный интерфейс для мобильных и веб-клиентов.

## Возможности

### Реализовано
- **Парсинг iCal формата** - преобразование данных из формата iCalendar в сущности занятий
- **Интеграция с MIREA API** - автоматическое получение актуального расписания
- **Многоуровневое хранение** - база данных → внешние источники
- **Поддержка всех сущностей** - группы, преподаватели, аудитории
- **Автоматическая синхронизация** - обновление ID из внешних источников
- **Типы занятий** - лекции (ЛК), практики (ПР), лабораторные (ЛАБ)
- **Повторяющиеся события** - поддержка recurring events через RRULE
- **Обработка ошибок** - централизованная обработка исключений
- **Оптимизированное сохранение** - пакетная обработка и параллельное выполнение

### Архитектура
```
Android Client → REST API → Schedule Service → [Database → External API]
```

## API Endpoints

#### Полное расписание (БД + API)
```http
GET /schedule/final/{titles}
```
**Параметры:**
- `titles` - список сущностей через двоеточие

**Ответ:** Полный список занятий с приоритетом: БД → внешние источники

## База данных

### Основные таблицы
- `lessons` - занятия
- `groups` - учебные группы
- `teachers` - преподаватели
- `rooms` - аудитории
- `lesson_group`, `lesson_teacher`, `lesson_room` - связи ManyToMany

## Оптимизация производительности

### Внедренные улучшения

**Пакетная обработка операций** - реализовано массовое сохранение сущностей через методы saveAll() вместо индивидуальных запросов, что сократило количество обращений к БД.

**Параллельная обработка справочников** - группы, преподаватели и аудитории обрабатываются одновременно в отдельных потоках через CompletableFuture.

**Устранение проблемы N+1 запросов** - заменены поштучные запросы на массовую загрузку данных с использованием оператора IN и сопоставлением в памяти.

### Результат оптимизации
Время обработки больших объемов данных сокращено в 10-50 раз за счет уменьшения количества запросов к базе данных с тысяч до единиц и параллельного выполнения независимых операций.

## Производительность

### Текущие метрики
- **Время ответа API**: ~150-300ms
- **Время парсинга iCal**: ~50-100ms на событие
- **Производительность БД**: оптимизирована для частых запросов
- **Память**: ~256MB RAM при стандартной нагрузке

### Мониторинг
- Логирование через SLF4J + Logback
- Метрики через Spring Boot Actuator
- Health checks на `/actuator/health`

## Обработка ошибок

Сервер возвращает стандартизированные ошибки:
```json
{
  "message": "Ошибка при получении данных из внешнего источника",
  "detailedMessage": "Connection timeout",
  "errorTime": "2024-01-15T12:00:00"
}
```

**Основные HTTP статусы:**
- `200` - успешный запрос
- `400` - некорректные параметры
- `404` - данные не найдены
- `503` - внешний API недоступен
- `500` - внутренняя ошибка сервера

## Процесс работы

1. **Получение запроса** от клиента с списком сущностей
2. **Валидация** типа и формата сущностей
3. **Поиск в БД** с проверкой актуальности
4. **Запрос к внешнему API** при отсутствии данных
5. **Парсинг iCal** и преобразование в сущности
6. **Сохранение в БД** с использованием оптимизированных методов
7. **Возврат результата** клиенту
